# Role Based UX Plan

## 1. 서비스 방향

Ringdu는 학원, 선생, 학생, 부모를 연결하는 무료 학원 관리 웹/앱 서비스다.

기본 학원 관리 기능은 무료로 제공한다. 다음 기능은 MVP 이후 유료화 또는 충전형 기능으로 확장할 수 있다.

```txt
카카오 알림톡 초과 발송
청구서 자동 발송
미납 자동 리마인드
고급 통계
학원 광고
프리미엄 커뮤니티 노출
추후 구독형 기능
```

## 2. MVP 역할 구조

프론트엔드의 role 기반 화면과 라우팅은 다음 5개 역할을 기준으로 한다.

| Role | 화면 표현 | 기본 홈 |
|---|---|---|
| ACADEMY | 학원 | `/academy` |
| TEACHER | 선생 | `/teacher` |
| STUDENT | 학생 | `/student` |
| PARENT | 부모 | `/parent` |
| ADMIN | 플랫폼 관리자 | `/admin` |

`OWNER`, `DESK`는 MVP에서 제외한다. 초기에는 `ACADEMY` 계정이 원장, 데스크, 운영자 역할을 통합해서 담당한다. 프론트 화면에서도 `OWNER`, `DESK`를 MVP 핵심 경로처럼 노출하지 않는다.

가입 정책:

```txt
ADMIN
- 일반 회원가입 불가
- 최초 계정은 개발/운영자가 생성
- local/dev 기본 계정은 admin@ringdu.com
- local/dev 기본 비밀번호는 ringdu1234
- 운영 환경에서는 기본 비밀번호를 그대로 사용하면 안 됨

ACADEMY
- 일반 회원가입 불가
- ADMIN이 /admin/academy-accounts/new 화면에서 생성

TEACHER/PARENT/STUDENT
- 일반 회원가입 가능
```

## 3. 역할별 UX 기준

### ACADEMY / 학원

학원 계정은 학원 운영 전체를 관리한다.

```txt
학원 정보 관리
선생 등록/초대
학생 등록/관리
부모-학생 연결 관리
클래스/수업 관리
시간표 관리
출석 관리
숙제 관리
공지 관리
청구서/수납 관리
학생 상담 기록 관리
```

### TEACHER / 선생

선생은 본인이 담당하는 수업과 학생을 관리한다.

```txt
담당 클래스 조회
담당 학생 조회
출석 요청 승인/반려
숙제 등록
수업 자료 공유
공지 작성
학생 상담 기록 작성
학부모 상담 내용 기록
```

### STUDENT / 학생

학생은 본인의 수업과 학습 정보를 확인하고 출석 요청을 한다.

```txt
내 클래스 조회
내 시간표 조회
출석 요청
내 출석 기록 조회
숙제 확인
학생 커뮤니티 이용
```

### PARENT / 부모

부모는 자녀의 학원 생활 정보를 확인한다.

```txt
자녀 등록 요청
자녀 출석 조회
자녀 시간표 조회
자녀 숙제 확인
청구서/수납 상태 확인
공개된 상담 내용 확인
학부모 커뮤니티 이용
```

### ADMIN / 플랫폼 관리자

ADMIN은 학원 내부 관리자가 아니라 Ringdu 플랫폼 운영자다.

```txt
전체 학원 관리
전체 사용자 관리
커뮤니티 신고 관리
광고 관리
서비스 공지 관리
유료 기능/충전 관리
운영 로그 확인
```

## 4. 상담 관리 UX

상담 관리는 신규 학생 유입 CRM이 아니다.

잘못된 방향:

```txt
상담 고객 등록
상담 후 등록 여부 결정
등록 완료 시 학생으로 전환
```

최신 방향:

```txt
상담 관리는 실제 재원생의 학생/학부모 상담 기록 관리 기능이다.
학원과 선생이 상담 기록을 작성한다.
부모는 부모 공개용 상담 내용만 확인한다.
```

공개 범위:

```txt
학원 내부용
선생 공유용
부모 공개용
```

내부 메모는 부모에게 보이지 않아야 한다.

## 5. 출석 UX

출석은 학생 요청 + 선생 승인 방식으로 설계한다.

```txt
학생이 해당 클래스에 들어감
→ 출석 버튼 클릭
→ 출석 요청 생성
→ 선생이 승인 또는 반려
→ 승인 시 출석 처리
```

출석 상태:

```txt
출석 요청
출석 완료
지각
결석
반려
취소
```

QR, 위치 인증, Wi-Fi 인증은 MVP에서 제외하고 후순위 확장 기능으로 둔다.

## 6. 청구서/수납 UX

초기에는 실제 결제 연동보다 청구서와 수납 상태 관리부터 구현한다.

```txt
학원이 청구서 생성
→ 부모가 청구서 확인
→ 학원이 수납 상태 변경
→ 부모가 납부 상태 확인
```

청구서 상태:

```txt
청구됨
납부 완료
미납
취소
```

카카오 알림톡 청구서 발송, 미납 자동 알림, PG 결제 연동은 MVP 이후 확장한다.

## 7. 커뮤니티와 광고

커뮤니티는 학생 커뮤니티와 학부모 커뮤니티로 나눈다.

학생 커뮤니티:

```txt
숙제 공유
질문 게시판
공부 자료 공유
시험 정보 공유
클래스별 게시판
자유 게시판
```

학부모 커뮤니티:

```txt
지역별 학원 정보 공유
학원 후기
교육 정보 공유
자녀 학습 고민
학원 추천
질문 게시판
```

광고는 학부모 커뮤니티와 연결한다. 지역별 학원 광고, 과목별 학원 광고, 커뮤니티 상단 배너, 추천 학원 노출은 MVP 이후 구현한다.

## 8. 개발 우선순위

현재 인증/인가 기본 구조와 프론트 role guard가 진행된 상태이지만, 역할 기획이 변경되었으므로 다음 개발 작업은 role 구조를 최신 기획에 맞게 정리하는 것이다.

```txt
1. Role 구조 정리
2. 학원 도메인
3. 선생/학생/부모 연결 구조
4. 클래스/수업 도메인
5. 출석 도메인
6. 학생 관리
7. 상담 관리
8. 숙제 도메인
9. 청구서/수납
10. 커뮤니티
11. 광고
12. 알림/유료화
```

```txt
feat/academy-domain
```

그다음 개발 작업:

```txt
feat/academy-domain
```
